## Findings & fixes

### Fixed

1. **[Nice-to-have → N3] No automated parser coverage for the shipped formats.**
   The only tests were env-config and path-validation. The 7 format parsers had
   zero happy-path coverage; every decryptor change since Session 2 was verified
   only by ad-hoc manual testing.
   *Fix (`653e5bf`, `test:`)*: `backend/tests/test_sample_parsing.py` parametrizes
   over all 32 real samples under `assets/configs/**` (HC×13, EHI×6, ZIV×6,
   DARK×4, TLS×2) and asserts the full detect→route→decrypt→normalize pipeline
   returns a `NormalizedConfig`, the detected format matches the file extension,
   and the filename survives normalization (guarding the Session 7 blank-filename
   regression). Decrypt success is intentionally not asserted (rotated/proprietary
   keys make "locked" a valid outcome). 32 passed.

2. **[Low] `/api/formats` returned stale capability data.**
   The endpoint advertised HC schemes `A1-A4` (missing `A5`, the v2.7+ ChaCha20
   scheme shipped Session 6), EHI `B1` only with `encrypted:false` (missing `B2`,
   the v6.3+ Argon2id scheme shipped Session 9; EHI configs are encrypted), and
   omitted the `ziv` (H1) and `lnk` formats entirely. Any client using this
   endpoint to discover capabilities got a wrong answer.
   *Fix (`308d360`, `fix(api):`)*: brought the list in line with `FormatEnum` +
   the router's `FORMAT_SCHEMES`. Verified live: `ehi[B2,B1]`, `hc[A5,A1-A4]`,
   `ziv[H1]`, `lnk[]` now reported. (Endpoint is exposed via `getFormats()` in
   `api.js` but not currently called by the UI — still a public API defect.)

3. **[Low] Misleading security comment on the notes iframe.**
   `renderNotesSandboxed` claimed srcdoc iframes are "already origin-isolated"
   and "can't access the parent." That is false — a srcdoc iframe inherits the
   embedder's origin (`file://`). The actual protection against a malicious
   notes payload (raw HTML from an untrusted config author, rendered after
   decryption) is the **inherited CSP** from `index.html`: `script-src 'self'`
   with no `unsafe-inline` blocks inline `<script>`/`onerror=`, and
   `img-src`/`connect-src` pin network egress to loopback. The srcdoc frame
   inherits that CSP, so the exploit is closed — but not for the reason the
   comment stated.
   *Fix (`289cd39`, `docs(ui):`)*: corrected the comment to describe the real
   protection and flag the sandbox-attr hardening. No runtime change.

### Flagged (not fixed)

- **[Low → N8] Notes iframe has no `sandbox` attribute (defense-in-depth).**
  The inherited CSP closes the actual hole, but a `sandbox="allow-same-origin"`
  attr (scripts disabled, same-origin retained so the auto-resize
  `contentDocument` read still works) would add a second, CSP-independent layer.
  Session 8 reported that adding a sandbox attr "prevented rendering" — likely a
  misdiagnosis (multiple rendering bugs were fixed in that commit), but this
  needs a run in the real Electron app to confirm styled notes still render
  before shipping. The in-app Chromium browser tool timed out this session, so
  it could not be verified here. Deferred to a session that can drive the GUI.

- Pre-existing backlog items **N1, N2, N4, N5, N6** unchanged — see
  `tasks/backlog.md`. N5 (unbounded `config_store`) and N6 (bare excepts,
  loose detector substring matches, `_ir_to_response` cosmetic inconsistency)
  need design decisions, not one-liners.

## Security posture (re-verified)

- **Path traversal (ADR-1/ADR-5):** `_validate_config_path` still applies the
  extension allowlist to BOTH the caller-supplied path and the resolved symlink
  target. `test_path_validation.py` (8 cases) passes, including the
  symlink-with-allowed-extension bypass. Intact.
- **CORS (ADR-2):** origins still pinned to loopback + `file://`/`null`,
  `allow_credentials=False`. Not reverted.
- **Dangerous patterns:** grep for `subprocess|os.system|eval|exec|pickle|__import__|shell=True|yaml.load` across `backend/` — none. The decryptors are pure crypto (pycryptodome/argon2), no shell-out, no deserialization of untrusted input.
