| Format | Samples | Decode status | Scheme |
|--------|---------|---------------|--------|

EHI extracts `proxy_host`, `payload`, and `sni` correctly (verified by

`host`/`server` field, so a card keyed on `host` looks empty), not a
decryption failure. Flagged as backlog N9.

## What was fixed

**TLS base64 padding bug** (`fix(decrypt)` `374c797`). Newer TLS Tunnel
builds emit `<base64>:::::` and strip the `=` padding. After the code
stripped the trailing colons, the surviving base64 length was not a
multiple of 4, so `base64.b64decode` raised `Incorrect padding` and the
decryptor returned `FAILED` **before the key loop ran** — an empty
decrypt trace with zero attempts, indistinguishable from "unsupported
format." Now re-pads to a multiple of 4 before decoding; the trace
honestly records the AES-GCM MAC failure against the known key. Verified:
the trace now shows a real `F1 / fail` attempt; HC 13/13 and EHI 6/6
still decode; 41/41 tests pass.

**Stray file cleanup** (`chore` `71f9d0a`). Removed
`assets/configs/hc/universal-kickoff.md` — a copy of the `.context`
protocol kickoff template accidentally committed into the HC samples
directory (commits `f1abf6a`/`00b493c`). Not referenced by any code.

## What could NOT be fixed, and why (researched online)

Both remaining encrypted formats are blocked on **secret material that
was rotated in newer app builds and is not publicly reversed** — not on
wrong algorithms. The algorithms in the codebase are correct and match
the authoritative public decryptors.

- **TLS Tunnel (F1).** Algorithm = AES-256-GCM, `IV(12) | ct | MAC(16)`,
  static base64 key. Confirmed against PANCHO7532's `HCDecryptor`
  (`lib/methods/tlsDecryptor.lib.js` + `cfg/keyFile.json`, on GitLab).
  InjectX already carries the **only** public TLS key
  (`VCTCp8Kq…`); it MAC-fails on the bundled samples. Upstream even
  *rejects* the `<base64>:::::` shape as "Non-Valid File: OOB" and the
  author warns the module "may not work for all versions" — i.e. these
  samples are a newer TLS build whose key isn't public.
- **ZIVPN (H1).** Algorithm = PBKDF2-HMAC-SHA256 (default 16-byte key →
  AES-128) + AES-GCM over `salt(32).nonce(12).ct+MAC`. Confirmed
  against KMKZ/EstebanZxx `X-Tools` `Main.py`, which uses password
  `fubvx788b46v`. InjectX's `ziv_decrypt.py` matches that reference
  byte-for-byte and already includes `fubvx788b46v`. A sweep of 16
  candidate passwords × {count 1/1k/10k/100k} × {dkLen 16/32} ×
  {SHA-256/SHA-1} = 256 combinations MAC-fails on every sample. The
  password was rotated in the build that produced these files.
- **DARK Tunnel.** Proprietary, in-app-only encryption; no public
  decryptor exists. Unchanged — correctly reports `NO_DECRYPTOR`.

## HA Tunnel (HAT) — answer to the user's question

Yes, HAT was wired up (Session 4 per the architecture notes; decryptor +
parser + scheme are present now):

- `backend/decrypt/hat_decrypt.py` — scheme **E1**, AES-128-ECB, keys
  from `KeyStore.aot` (2 keys), validates output starts with `{"`.
  Handles the three known HA Tunnel Plus JSON shapes (`profile`,
  `profilev4`, `configuration`). Algorithm matches HCDecryptor's
  `aotDecryptor.lib.js`.
- `backend/parser/hat_parser.py`, `FormatEnum.HAT`, `SchemeEnum.E1`,
  router entry, and `.hat`/`.ha` in `ALLOWED_EXTENSIONS` all exist.

**But it is unverified end-to-end: there are zero `.hat` sample files**
(`assets/configs/hat/` holds only `.gitkeep`). We cannot claim HAT
"works" until a real `.hat` config is dropped in and decoded. Backlog
N10.
