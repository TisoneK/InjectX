## 2. What was built

### HTML (`frontend/index.html`)

New `<div class="sni-defensive-panel">` section inside `view-snihunter`, after
the `.sni-layout` two-column grid. Contains:

- A panel title with an ADR-9 reminder ("verify SNI/Host enforcement · ADR-9").
- An intro block explaining what the probe does, with inline `bypassable` /
  `enforced` tags color-coded to match the verdict banner.
- A 3-column form (SNI input | Host input | PROBE button).
- A result container (`#sni-fronting-result`) that starts hidden and is filled
  by `renderSniFrontingResult()`.

### Renderer (`frontend/src/scripts/renderer.js`)

- `state.sni.fronting` — holds the last `SniFrontingResult` so navigating away
  and back preserves it (same pattern as `state.sni.currentJob`).
- `SNIFRONTING_VERDICT_META` — verdict → {label, cls, blurb} map. `bypassable`
  → danger (red), `enforced` → ok (green), `indeterminate` → warn (amber),
  `error` → muted. Each verdict gets a one-sentence plain-language blurb.
- `sniRunFronting()` — reads the two inputs, shows a "PROBING..." state, calls
  `API.sni.fronting(sni, host)`, hands the result to the renderer, shows a
  toast + logs to the activity console.
- `renderSniFrontingResult(r)` — renders: (1) a verdict banner with the pill +
  blurb, (2) a 2-column detail grid with all 11 fields the backend returns
  (SNI→Host, HTTP status, Server header, TLS handshake, both cert CNs,
  cert-changes-with-SNI, host-covered-by-cert, DNS consistency, target IP,
  elapsed), (3) the backend's `notes` array as a bulleted list.
- `renderSniHunterView()` updated to re-render the last fronting result when
  navigating back to the view.
- `setupSniHunter()` — wires the PROBE button + Enter-key on both inputs.

### CSS (`frontend/src/styles/main.css`)

~110 new lines: `.sni-defensive-panel`, `.sni-defensive-intro` (with a violet
left border to visually distinguish the defensive surface from the offensive
scan surface), `.sni-defensive-form` (3-column grid that stacks on narrow
windows), `.sni-fronting-btn` (violet gradient to match the defensive framing),
`.sni-fronting-verdict` banner with the 4 verdict color variants, `.sni-fronting-detail`
2-column grid, `.sni-fronting-notes`. Uses the existing CIPHER_OPS variables.

## 5. Design notes

- **Why a separate panel, not a per-result-row "FRONT?" button.** The N17
  backlog entry suggested "a per-result-row FRONT? action that runs the probe
  against the selected config's current SNI as host." I considered this and
  chose the separate-panel approach instead, because:
  - ADR-9 frames the fronting probe as a distinct *defensive* capability
    ("verify enforcement"), not an offensive scan action. Visually separating
    it (below the scan surface, with a violet accent vs the scan surface's
    green) reinforces that framing.
  - The probe takes two free-form hostnames (SNI + Host), not just one. A
    per-result-row button would have to pick one of them implicitly, which is
    lossy. The panel lets the user enter any pair.
  - The panel matches the terminal command's shape (`sni fronting <sni> <host>`)
    exactly — same two inputs, same output fields.
  - If a per-result-row shortcut is wanted later, it can be added as a fifth
    action button that pre-fills the panel's SNI field from the result row's
    hostname and scrolls to it. Deferred — the panel is the primary surface.

- **Verdict color mapping.** `bypassable` → red (danger) because it means the
  filter *leaks* — that's the "bad" outcome from a defensive standpoint.
  `enforced` → green (ok) because the cross-check holds. This is the opposite
  of the offensive scan's `working` → green, which is intentional: the two
  surfaces answer mirror questions, and the colors reflect each surface's own
  "good" outcome.
