## 1. Executive Summary

This was the first `.context/` session on the InjectX repo. InjectX is a desktop application (Electron frontend + Python FastAPI backend) for inspecting and decrypting VPN tunnel configuration files produced by Android tunneling apps (.ehi, .hc, .hat, .dark, .tls, .npv4, .nsh, .vhd, .ovpn).

The session discovered and fixed a **critical path-traversal vulnerability** in the `/api/config/parse` and `/api/config/detect` HTTP endpoints: any local process could read arbitrary files (e.g., `/etc/passwd`, `~/.ssh/id_rsa`) by calling the loopback API. The fix adds an extension allowlist, path resolution, and a 50 MiB size cap at the API boundary. Three additional security hardening commits tighten CORS, cap upload size, and validate upload extensions.

Beyond security, the session fixed a latent routing bug (`/api/config/export` and `/api/config/detect` were shadowed by `/api/config/{config_id}` and silently 404'd from the frontend), migrated 8 Pydantic v2 deprecation warnings, replaced `print()` with proper `logging`, and synced the stale `package-lock.json` to `package.json` v0.4.0.

Several architectural items remain open (unbounded in-memory config store, no test infrastructure, stale architecture doc) and are flagged in §6 for future sessions.

## 4. Findings (by severity)

### Critical

#### C1 — Path traversal in `/api/config/parse` and `/api/config/detect`

- **Severity:** Critical
- **Description:** The `/api/config/parse?filepath=...` and `/api/config/detect?filepath=...` endpoints accepted an arbitrary absolute filepath and read the file at that path with no validation. The backend binds to `127.0.0.1` only, but any local process (other apps, malicious scripts, browser-based CSRF given the prior `allow_origins=["*"]`) could call the HTTP endpoint and read arbitrary files.
- **Impact:** Verified during baseline: `curl "http://127.0.0.1:8742/api/config/parse?filepath=/etc/passwd"` returned `200 OK` with the parsed `/etc/passwd` content (format detected as `unknown`, file size and hex preview in the response). Any local file (`~/.ssh/id_rsa`, `~/.aws/credentials`, `/proc/self/environ`) was readable.
- **Recommendation:** Validate the filepath at the API boundary: (1) require absolute path, (2) extension allowlist (only `.ehi/.hc/.hat/...`), (3) `Path.resolve(strict=True)` to collapse `..` segments and reject broken symlinks, (4) file size cap. Apply the same validation to both endpoints via a shared `_validate_config_path` helper.
- **Status:** **Fixed** in `d0f8cd9`.
- **Related commit:** `d0f8cd9` `fix(security): close path-traversal in /api/config/parse and /api/config/detect`

### High

#### H1 — CORS `allow_origins=["*"]` with `allow_credentials=True`

- **Severity:** High
- **Description:** The backend's CORS middleware used `allow_origins=["*"]` together with `allow_credentials=True`. This combination is invalid per the CORS spec (browsers reject credentialed requests with wildcard origin). Even setting aside the spec violation, wildcard origins on a loopback-bind backend are too permissive — any local web page (e.g., a browser tab open to an attacker-controlled site) could issue cross-origin requests and read responses.
- **Impact:** A malicious local web page could read parsed config data, upload arbitrary files for parsing, or delete configs via the API. Combined with C1, it could read arbitrary files via the loopback.
- **Recommendation:** Restrict origins to the ones the Electron renderer actually uses: `file://` (Electron's default `loadFile` origin) and `http://127.0.0.1:8742` / `http://localhost:8742`. Disable `allow_credentials` (no cookies/auth used). Restrict methods to `GET/POST/DELETE` and headers to `Content-Type`.
- **Status:** **Fixed** in `0cb9e44`.

### Medium

#### M1 — `/api/config/upload` accepts any file extension + no size limit

- **Severity:** Medium
- **Description:** The upload endpoint saved files using `Path(file.filename).suffix` as the extension and `await file.read()` to load the entire file into memory, with no extension validation and no size cap. A hostile upload with extension `.exe` would land in `UPLOAD_DIR` and be passed through `parse_config`, which would attempt decryption with all 76+ HC keys on a file that's clearly not a config.
- **Impact:** Disk pollution, CPU burn on hostile uploads, potential OOM on very large uploads.
- **Recommendation:** Validate the extension against the same allowlist as `/parse`. Cap uploaded size at 50 MiB (config files are kilobyte-scale). Reject empty uploads.
- **Status:** **Fixed** in `2bf7337`.

#### M2 — `/api/config/export` and `/api/config/detect` shadowed by `/api/config/{config_id}` (latent frontend bug)

- **Severity:** Medium
- **Description:** FastAPI matches routes in registration order. `/api/config/{config_id}` (GET, registered at line 295) shadowed `/api/config/detect` (line 332) and `/api/config/export` (line 365), both of which were registered later. A GET request to either literal path matched the parameterized route instead, returning `404 "Config not found: detect"` (or `export`). The `/detect` shadowing was masked because the endpoint was `@app.post` and the renderer's `detectFormat` IPC handler uses default-fetch GET, so it silently 404'd — but the renderer never calls `detectFormat`, so nobody noticed. The `/export` shadowing became visible when `/export` was changed to GET to match the frontend's `exportConfig` call (which is wired to the Export button).
- **Impact:** The Export button in the detail view was broken. The detectFormat API was unreachable. Both are user-facing regressions (or latent bugs, depending on whether anyone tried to use them).
- **Recommendation:** Reorder route registrations so literal paths come before parameterized ones. Add a comment to `/detect` and `/export` explaining the ordering constraint so a future contributor doesn't re-introduce the bug.
- **Status:** **Fixed** in `2490ad3`.

### Low

#### L1 — Pydantic v2 `BaseModel.dict()` deprecation

- **Severity:** Low
- **Description:** 8 call sites used the deprecated `BaseModel.dict()` method (removed in Pydantic v3.0). Each emitted a `PydanticDeprecatedSince20` warning on every request that touched the IR.
- **Impact:** Log noise; will break on Pydantic v3 upgrade.
- **Recommendation:** Migrate to `BaseModel.model_dump()`.
- **Status:** **Fixed** in `547f8fe` (and partially in `2bf7337` during the upload hardening).

#### L2 — `print()` in production backend code

- **Severity:** Low
- **Description:** `main.py`'s lifespan startup/shutdown messages used `print()`, which writes to stdout without timestamps or severity, interleaved with uvicorn's structured logs.
- **Impact:** Hard to filter; no severity levels for future log calls.
- **Recommendation:** Switch to `logging.getLogger("injectx")` with `basicConfig` providing ISO-ish timestamps and level prefix.
- **Status:** **Fixed** in `68f21c7`.

#### L3 — `package-lock.json` stale (v0.2.0 vs `package.json` v0.4.0)

- **Severity:** Low
- **Description:** The lockfile's root version was `0.2.0` while `package.json` was at `0.4.0` — likely the lockfile was not committed after a prior `package.json` version bump.
- **Impact:** `npm ci` fails when the lockfile's root version doesn't match `package.json`. Anyone running fresh install would have to use `npm install` instead.
- **Recommendation:** Commit the regenerated lockfile.
- **Status:** **Fixed** in `8e1e00d`.

#### N3 — No test infrastructure

- **Severity:** Nice to Have (process)
- **Description:** No `tests/` directory, no `conftest.py`, no `pyproject.toml [tool.pytest]`, no test runner config. The 7 format parsers and 7 decryptors have no automated coverage — changes to crypto code are currently verified only by ad-hoc manual testing.
- **Impact:** Regressions in decryption logic (e.g., a key rotation, an algorithm tweak) won't be caught automatically.
- **Recommendation:** Add `pytest` + `pytest-cov` to dev dependencies. Add a `tests/` directory with at least one happy-path test per format (using a known-good sample file) and one negative test per format (malformed input). Add `ruff` and `mypy` configs. This is a significant setup effort; flagged for a future session.

#### N4 — No CI

- **Severity:** Nice to Have (process)
- **Description:** No `.github/workflows/` directory. Pushes to `main` are not validated.
- **Impact:** Broken code can land on `main` undiscovered.
- **Recommendation:** Add a minimal GitHub Actions workflow that runs `pytest`, `ruff check`, and `npm install` on every push/PR. Depends on N3.

#### N5 — Unbounded in-memory `config_store`

- **Severity:** Nice to Have (architecture)
- **Description:** `config_store: dict[str, dict] = {}` grows forever as configs are uploaded/parsed. There's no eviction, no max size, no LRU. Each entry includes the full `decrypt_trace` (which can be large — 21 attempts × multiple fields for a B1 scheme that fails on all keys).
- **Impact:** Long-running desktop sessions that parse many configs will accumulate memory. Not a crash risk for typical usage (dozens of configs), but a power user parsing hundreds could see noticeable memory growth.
- **Recommendation:** Either (a) cap `config_store` at N entries with LRU eviction, or (b) persist to SQLite (per architecture doc §10.2 — already planned for v2). Flagged as architectural; not auto-fixed.

#### N6 — Bare `except Exception:` clauses throughout decrypt code

- **Severity:** Nice to Have (code quality)
- **Description:** 60+ bare `except Exception:` clauses in `backend/decrypt/` and `backend/parser/` swallow errors silently. In the decrypt code this is somewhat intentional (try-many-keys pattern — one key failing shouldn't abort the loop), but in the parser code it hides bugs.
- **Impact:** Silent failures make debugging difficult. A typo in a field name, a type mismatch, or a malformed sample file produces no error output.
- **Recommendation:** In decrypt code, leave as-is (intentional) but log at `DEBUG` level. In parser code, narrow the exception type (e.g., `except (KeyError, TypeError, json.JSONDecodeError):`) and log at `WARNING`. This is a broad refactor; flagged for a future session.

#### N7 — Documented env vars not implemented

- **Severity:** Nice to Have (features)
- **Description:** Architecture doc §11 documents `INJECTX_PORT`, `INJECTX_HOST`, `INJECTX_UPLOAD_DIR`, `NODE_ENV` env vars. The backend code reads none of them — `host` and `port` are hardcoded in `main.py` (`127.0.0.1:8742`), `UPLOAD_DIR` is hardcoded to `~/.injectx/configs`. `NODE_ENV` is checked in `frontend/main.js` (for DevTools) — that one is implemented.
- **Impact:** Users can't override the port (useful if 8742 is taken) or the upload directory (useful for portable installations).
- **Recommendation:** Read env vars with sensible defaults in `main.py`: `INJECTX_PORT = int(os.environ.get("INJECTX_PORT", "8742"))`, etc. Small change; could be done in a future session.
